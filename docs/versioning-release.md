# バージョニングとリリース運用

このリポジトリのバージョン付与・リリース記録・公開ロードマップ更新のルールを定義します。
バージョンに関わる判断はこのファイルを source of truth とします。

## バージョニング方針（SemVer）

[Semantic Versioning 2.0.0](https://semver.org/lang/ja/) に従い、`MAJOR.MINOR.PATCH` で管理します。
**変更は細かく拾い、リリース単位ごとに 1 段ずつ番号を上げます**（増分は最大 +1）。

- **MAJOR（+1.0.0）**: 世代刷新・大規模リニューアル、または後方非互換となる基盤変更。
  - 上げると MINOR と PATCH は 0 にリセットする。
  - 例: サイト UI の全面刷新、Next.js のメジャーアップグレード、CMS 移行。
- **MINOR（+0.1.0）**: 後方互換を保つ新機能・新ページの追加。
  - 上げると PATCH は 0 にリセットする。
  - 例: サイト内検索、世界地図、FAQ ページ、地域階層検索。
- **PATCH（+0.0.1）**: バグ修正・小さな改善・パフォーマンス/デザインの微調整・SEO 調整・リファクタリング・依存更新。
  - 例: 表示崩れ修正、フォント最適化、`chore(deps)`、内部リファクタリング。

### 採番の粒度

- 1 つのリリース（PR / まとまった作業）= 1 バージョン。複数日にまたがる細かな修正は、まとめて 1 つの PATCH にしてよい。
- 番号は飛ばさず、常に直前の版から 1 段だけ上げる（例: `2.4.0` の次の機能追加は `2.5.0`、その後の修正は `2.5.1`）。
- 純粋なコンテンツ追加（記事・写真）やデプロイ再試行などは採番しない。

### 単一の真実

- 正となるバージョンは **`package.json` の `version`** です。
- 公開用の更新履歴（`src/components/features/roadmap/UpdateList.tsx` の `UPDATES`）と
  `CHANGELOG.md` の最新版は、常に `package.json` と一致させます。

### 更新履歴の種別（UpdateList の `type`）

公開更新履歴の種別は **バージョン番号の桁で機械的に決める**（恣意的に選ばない）。

| `type`    | 表示ラベル           | 条件                | 例                                   |
| --------- | -------------------- | ------------------- | ------------------------------------ |
| `major`   | メジャーアップデート | `x.0.0`             | 世代刷新・全面リニューアル・基盤刷新 |
| `feature` | 機能アップデート     | `x.Y.0`（Y>0, Z=0） | 新機能・新ページ                     |
| `patch`   | パッチアップデート   | `x.y.Z`（Z>0）      | 修正・小改善・パフォーマンス調整など |

- メジャーは節目（`x.0.0`）のみ。実質的にほとんど発生しない。
- 公開更新履歴の文面は **利用者目線**で書く。実装の内部用語（バンドル・preconnect・woff2・
  canonical・スキーマ名など）は避け、「何がどう良くなったか」を簡潔に書く。
  技術的な詳細は `CHANGELOG.md` 側に残す。
- 実在しないバージョンを更新履歴に新設しない（`package.json` ↔ `CHANGELOG` ↔ `UpdateList` の整合を保つ）。

## いつバージョンを上げるか

- 利用者から見える変更（機能・UI・パフォーマンス・SEO・修正）を含む PR をマージするときに上げます。
- 純粋な内部変更（リファクタリング・ドキュメント・CI 設定のみ）は原則バージョンを動かしません。
  - ただし利用者に体感差が出る大規模リファクタリングは MINOR とします。
- `chore(deps)` 単独の更新は、脆弱性修正など影響が大きい場合のみ PATCH を上げます。

## リリース手順

1. 変更種別から SemVer の増分（MAJOR/MINOR/PATCH）を決める。
2. `package.json` の `version` を更新する。
3. `CHANGELOG.md` の先頭に新しい版の見出しを追加し、`Added / Changed / Fixed / Performance / Security` 等で分類して記載する。
4. 利用者に告知したい内容であれば、`UpdateList.tsx` の `UPDATES` 先頭に 1 件追加する。
   - 直前まで `isNew: true` だったエントリから `isNew` を外し、新しいエントリに `isNew: true` を付ける。
   - `date` は `YYYY.MM.DD`、`version` は `vX.Y.Z` 形式。
5. 必要に応じて `src/data/roadmap.tsx` のステータスを実態に合わせて更新する（下記）。
6. `pnpm exec tsc --noEmit` と `pnpm lint`、`pnpm format:check` で検証する。

## 提供終了（削除）した機能の扱い

一度公開して後から削除した機能は、履歴から消さずに「提供終了」と分かる形で残します。

- **公開時のエントリを保持**し、`UpdateList.tsx` の該当エントリに次を付ける。
  - `removed: true`
  - `removedNote: "YYYY年M月に提供を終了しました。"`（終了時期と必要なら理由）
  - UI 側では、提供終了エントリはカードを淡色化し、タイトルに打ち消し線、「提供終了」バッジと補足を表示する。
- **`CHANGELOG.md`** には、削除を行ったリリースの `### Removed` に具体的な対象を記載し、
  公開時のリリースには「後に削除」の注記を添える。
- 例: Generative UI（`4.3.0` で公開 → `4.6.0` で削除）、Focus Mode（`3.7.0` で公開 → 2026-04 に削除）、
  PWA オフライン対応（`2.4.0` で公開 → `4.6.0` で削除）。

## 公開ロードマップの更新（`src/data/roadmap.tsx`）

`ROADMAP_DATA` はサイトの `/roadmap` に表示される将来計画です。実態と乖離させないこと。

ステータスの意味:

| status           | 表示ラベル        | 意味                                       |
| ---------------- | ----------------- | ------------------------------------------ |
| `COMPLETED`      | Check-in Complete | 完了・安定稼働している過去のマイルストーン |
| `DEPLOYED`       | In Flight         | 公開済みだが継続的に育てている段階         |
| `CURRENT_TARGET` | Boarding Now      | 現在最優先で取り組んでいる対象（1 件）     |
| `PENDING`        | Scheduled         | 着手予定                                   |
| `CONCEPT`        | Planning          | 構想段階                                   |

更新ルール:

- `CURRENT_TARGET` は原則 **1 件のみ**。四半期が進んだら次のマイルストーンへ移す。
- 完了したマイルストーンは `COMPLETED`（または継続育成中なら `DEPLOYED`）に更新する。
- `season` は実態の四半期（`YYYY.Qn`）に合わせる。

## 関連ファイル

- `package.json` — 正のバージョン番号。
- `CHANGELOG.md` — 開発者向けの詳細な変更履歴。
- `src/components/features/roadmap/UpdateList.tsx` — 利用者向けの公開更新履歴。
- `src/data/roadmap.tsx` — 公開ロードマップ（将来計画）。
